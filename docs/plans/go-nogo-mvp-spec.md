# Go / No-Go 上线前检查 — MVP 规格（收敛版）

面向：**一人公司 / 资源极少的团队**。  
产品只做一件事：**判断当前版本是否建议上线（Go / No-Go）**，并标明 **上线前必须改** vs **可上线后优化** vs **非关键**。

---

## 一、原则

- 所有功能必须服务 **Go / No-Go 结论**；不服务的功能 **不做**。
- 第一版 **不做**：团队协作、历史记录、复杂仪表盘、实时监控、全语言铺开。
- 使用场景刻意收窄：**发版前 / 大版本前** 那一次检查（像体检，不是天天扫）。

---

## 二、MVP 仅 5 个检测模块（不加第六个）

| # | 模块 | 范围（刻意收窄） |
|---|------|------------------|
| 1 | **安全关键项** | 明显未鉴权接口（启发式）、硬编码密钥/token（密钥扫描）、**高危**依赖漏洞（阈值过滤，不做全量 CVE 罗列） |
| 2 | **环境 & 配置** | debug 开启、`.env` 暴露风险、CORS 过宽、API base URL 异常等（静态/配置模式匹配 + 少量规则） |
| 3 | **基本可用性（轻）** | 明显错误路径/500 线索（启发式）、核心流程 **不** 做复杂 E2E，只做「明显不能用」信号 |
| 4 | **依赖 & 版本** | known CVE（接现有库/扫描器）、deprecated 包（规则/元数据，能接则接） |
| 5 | **发布 Checklist（AI）** | 非检测：LLM 生成「可能遗漏项」（隐私页、监控、日志脱敏等），**无法律结论** |

---

## 三、报告页面结构（4 块，顺序固定）

1. **结论（置顶，3 秒内可读）**  
   - `不建议上线 (No-Go)` 或 `可以上线 (Go)`  
   - **原因** 或 **建议优化** 各 1～3 条人话 bullet  

2. **Top 3 问题（强制最多 3 条、排序）**  
   - 标题如「未鉴权接口」「CORS 全开放」「高危依赖」  

3. **详细报告（可折叠）**  
   - 分：**安全 / 配置 / 依赖 / 可用性**  
   - 每条：**是什么 → 为何严重 → 怎么修**（尽量含代码/配置片段）  

4. **上线建议（决策辅助）**  
   - **必须修复后上线**  
   - **可上线后优化**  
   - 非关键可归入「可忽略 / 低优先级」类（与 Critical 区分）

---

## 四、用户流程（仅 3 步）

1. **输入**：Git 仓库 URL（GitHub / Gitee 等 HTTPS）**或** 上传 zip（第一版可二选一或分阶段）  
2. **一个按钮**：「检查是否可以上线」  
3. **报告页**：无注册、无复杂配置（前期）

---

## 五、一句话产品描述（Landing 可用）

**中文：**  
上线前最后一次把关：一眼看清 **能不能发**，以及 **必须先改什么**。

**英文：**  
Ship with confidence — one clear **Go / No-Go** before release, with **must-fix** vs **later** items.

---

## 六、与当前代码库（Prelaunch）的关系

当前 Prelaunch 已有：多工具扫描、归一化、分维度报告、LLM 摘要/清单。  
要完全贴合本规格，还需：**Go/No-Go 裁决规则**、**仅 Top3**、**配置/可用性专用规则层**、**报告四区块重构**、可选 **zip 上传**。详见同目录后续实现任务拆解（可单独开 `go-nogo-implementation.md`）。
